# Task Manager API

A full production-style Task Manager REST API built with Spring Boot 3, Spring Security, JWT, Spring Data JPA, and PostgreSQL.

## Features
- **REST API** for managing tasks.
- **JWT Authentication** (Login/Register).
- **Role-based Access Control** (Users can manage their tasks; Admins can manage all tasks).
- **Clean Layered Architecture** (Controller, Service, Repository, Entity, DTO).
- **Global Exception Handling** and Input Validation.
- **Dockerized Deployment** (Application + PostgreSQL database).

## Tech Stack
- Java 21
- Spring Boot 3.2.x (Web, Data JPA, Security, Validation)
- PostgreSQL
- JWT (jjwt)
- Maven
- Docker & Docker Compose

## Getting Started

### Prerequisites
- Docker and Docker Compose installed
- OR Java 21 and PostgreSQL installed locally

### Run with Docker Compose
To run the entire stack (PostgreSQL + Spring Boot application) using Docker:

```bash
docker-compose up --build
```
The API will be available at `http://localhost:8080`.

### Endpoints

#### Authentication
- `POST /api/auth/register` - Register a new user (Body: `{"username": "user", "password": "password", "role": "USER"}`)
- `POST /api/auth/login` - Login and get JWT token (Body: `{"username": "user", "password": "password"}`)

#### Tasks (Requires `Authorization: Bearer <token>`)
- `POST /api/tasks` - Create a task
- `GET /api/tasks` - Get tasks (Returns user's tasks or all tasks for Admin)
- `GET /api/tasks/{id}` - Get a specific task
- `PUT /api/tasks/{id}/status?status=IN_PROGRESS` - Update a task status
- `DELETE /api/tasks/{id}` - Delete a task
- `GET /api/tasks/admin/all` - Get all tasks (Requires ADMIN role)

## The Maven Build Lifecycle

Maven operates on a sequential build lifecycle. When you execute a phase (e.g., `mvn install`), Maven automatically runs every phase preceding it. In this project, we explicitly demonstrate each step.

### Core Phases:
1. **`mvn clean`:** Removes all files generated by previous builds (deletes the `target/` directory).
2. **`mvn validate`:** Verifies that the project structure is correct and all required information (like dependencies) is available.
3. **`mvn compile`:** Translates `.java` source files in `src/main/java` into Java bytecode (`.class` files).
4. **`mvn test`:** Executes the unit tests in `src/test/java` using the Surefire plugin. *JaCoCo calculates our coverage percentage simultaneously.*
5. **`mvn package`:** Bundles the compiled code into distributable formats. This creates the Spring Boot executable fat JAR containing both our compiled code and all required dependencies.
6. **`mvn verify`:** Runs checks on the results of integration tests (or other quality criteria) to ensure the project meets defined standards.
7. **`mvn install`:** Places the generated JARs into your local `.m2` repository, making them available as a dependency for your other local Java projects.
8. **`mvn deploy`:** *(Commented out by default)* Copies the final package to a remote repository (like Nexus or Artifactory) for sharing. Requires `<distributionManagement>` configuration in the `pom.xml`.

## CI/CD Automation Scripts

To automate the build process and prevent manual typing, the project includes two automation scripts:
- `build.sh` (for Linux/macOS/Git Bash)
- `build.ps1` (for Windows PowerShell)

These scripts execute the Maven lifecycle step-by-step. They utilize strict error handling (`set -e` or `$ErrorActionPreference = "Stop"`) to ensure the build halts immediately if any phase fails.

### Code Coverage Report
After the script runs `mvn test`, JaCoCo generates a human-readable HTML site.
You can view your code coverage by opening:
`target/site/jacoco/index.html`

## Native Binary Distribution (`jpackage`)

After completing the standard Maven lifecycle, our build scripts execute a bonus CI step: Generating a Native Application Image.

Using the `jpackage` utility (included in JDK 14+), we bundle our fat JAR alongside a lightweight, optimized JVM runtime. 

```bash
jpackage --type app-image --name <app.name> --input target/jpackage-input/ --main-jar <artifactId>-<version>.jar --dest target/native-image --win-console
```
*(The build scripts extract these values dynamically from `pom.xml` using `mvn help:evaluate`)*

- `--type app-image`: Creates a standalone directory containing the executable and its own isolated Java runtime (no system installation of Java required to run it).
- `--win-console`: Specific to Windows, it forces the application to open a terminal output.
- Note: A specific `--input` directory (`target/jpackage-input/`) is utilized to prevent `jpackage` from recursively bundling its own output folder.

### Running the Native Application
Once the build completes, a platform-specific binary is generated.

**On Windows:**
```powershell
.\target\native-image\<app.name>\<app.name>.exe
```

## Setup and Usage Without Docker

**Prerequisites:**
- Java Development Kit (JDK) 21 installed and on your system `PATH`.
- Apache Maven installed and on your system `PATH`.

**To build the project:**

On Windows:
```powershell
.\build.ps1
```
*(If you encounter execution policy errors, run: `powershell.exe -ExecutionPolicy Bypass -File .\build.ps1`)*

On Linux/macOS:
```bash
chmod +x build.sh
./build.sh
```
